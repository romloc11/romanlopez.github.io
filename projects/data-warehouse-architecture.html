SQL Data Warehouse for Accounts Receivable Analytics

Business Context

The Accounts Receivable team frequently needs to analyze customer payment behavior, credit risk, and order processing metrics in order to support operational decisions. However, the required data was distributed across multiple operational systems and transactional databases.

As a result, analysts were forced to manually gather information from several sources before performing any analysis.

Problem

Before this project, there was no centralized data repository designed for analytics. Every time analysts needed to perform an analysis, they had to repeat the same process:

  • Access multiple operational systems.
  • Export raw data into several Excel files.
  • Clean and standardize the data manually.
  • Merge datasets using spreadsheet formulas.
  • Build temporary reports for decision making.

This workflow created several challenges:

  • Significant time spent on data preparation instead of analysis.
  • High risk of manual errors.
  • Difficulty maintaining consistent metrics.
  • Inability to generate automated reports.
  • Limited scalability for future analytics initiatives.

Project Objectives

The goal of this project was to design and implement a centralized Data Warehouse that consolidates data from operational systems into a structured architecture optimized for analytical workloads.

  • Create a reliable and scalable data architecture.
  • Centralize business data used by analysts.
  • Standardize transformations and business logic.
  • Enable direct SQL analysis.
  • Support automated reporting with Power BI.
  • Provide a foundation for future advanced analytics.

Data Warehouse Architecture

The Data Warehouse was designed following the Medallion Architecture, which organizes data processing into three logical layers: Bronze, Silver, and Gold. This architecture improves data reliability, maintainability, and scalability.

  • Bronze Layer – Stores raw data extracted from operational systems. Data remains as close as possible to the source format to preserve traceability.
  • Silver Layer – Applies data cleansing, normalization, and business rules. In this layer, inconsistencies are resolved and data structures are standardized for analytical usage.
  • Gold Layer – Contains curated datasets modeled specifically for analytics and reporting tools such as Power BI.

Dimensional Data Model (Star Schema)

The analytical layer of the Data Warehouse was implemented using a Star Schema, a widely used dimensional modeling approach that optimizes query performance and simplifies data exploration.

In this model:

  • Fact tables store measurable business events such as transactions, orders, or financial values.
  • Dimension tables provide contextual attributes that describe those events, such as customers, dates, or products.

This structure allows analysts to perform complex queries efficiently and enables seamless integration with Business Intelligence tools.

ETL Process

Data ingestion and transformation processes were implemented using SQL-based ETL pipelines.

  • Extraction of transactional data from source systems.
  • Incremental loading of raw data into the Bronze layer.
  • Data cleansing and transformation processes applied in the Silver layer.
  • Construction of analytical fact and dimension tables in the Gold layer.
  • Scheduling of automated refresh processes.

This ETL structure ensures that data remains consistent, reproducible, and ready for analytical workloads.

Benefits and Impact

The implementation of the Data Warehouse significantly improved the organization’s analytical capabilities.

  • Analysts can query curated datasets directly using SQL.
  • Data preparation processes are centralized and reusable.
  • Power BI dashboards can connect directly to the Gold layer.
  • Reporting processes can be automated.
  • Data quality and consistency improved across analyses.
  • The architecture provides a foundation for future machine learning models.
